## What this is

Phase 0b is a **refactor** — no behavior change is intended. log-command.py was reduced from 3199 LOC to 243 LOC (bootstrap + main + dispatch only); all logic was moved verbatim (move-code methodology) into 13 focused modules under `hooks/scripts/cclogger/`. The package's `__init__.py` runs `_ensure_dazzle_filekit()` at import time so downstream `from dazzle_filekit import ...` calls succeed, and re-exports symbols for tests that look up attributes on the package namespace.

The automated suite + diff_check.py give high confidence that nothing structural broke. This checklist covers what those CAN'T cover: real plugin install/reload from the cache, live Claude Code event payloads against the actual transcript JSONL format, dazzle_filekit auto-install fallback in a fresh environment, the rich `"path:line ← snippet (-N/+ML)"` Edit/Write/Grep formatting that humans actually read, and cross-shell behavior on Windows (cmd.exe, PowerShell, bash/WSL).

## Prerequisites

1. **OS**: Windows 10/11, macOS, or Linux. Most tests apply to all; sections marked `[Windows-only]` skip on POSIX.
2. **Tooling**: Python 3.8+, `git`, Claude Code CLI installed and signed in.
3. **Working directory**: a checkout of this repo at the Phase 0b commit. The plugin marketplace must be configured to point at this local folder (verify with `cat ~/.claude/plugins/known_marketplaces.json` — should show `"source": "directory"` pointing at `<repo>/github/`).
4. **State isolation**: the live-session sections (2-7) actually create real files under `~/.claude/sesslogs/`. They use disposable session directories (named after this checklist), so they won't collide with real work — but be aware they will appear in your `~/.claude/sesslogs/` listing until you clean them up (Section 10).
5. **Reset between sections**: most sections are idempotent or self-cleaning. Where they're not, an explicit reset is called out in the section.

> **WARNING**: do NOT run this checklist while you have important uncommitted work in another Claude Code session. The hook writes to your real `~/.claude/sesslogs/` and `~/.claude/session-states/` for any session that triggers it. Fresh sessions in throwaway directories are the safest pattern.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact behavior the modularization could break. If any fail, the work is broken and you can stop here — the detailed sections below probably can't save you.

### HV.1 — Hook subprocess loads cleanly (no ImportError)

Validates: the bootstrap line (`sys.path.insert`) + cclogger package init + `_ensure_dazzle_filekit()` chain works when log-command.py is invoked as a subprocess (the way Claude Code actually invokes it).

- [ ] **Steps** (cross-shell safe — uses Python directly):

  ```
  python hooks/scripts/log-command.py < /dev/null 2>&1
  ```

  **Windows cmd.exe**:
  ```cmd
  echo {} | python hooks\scripts\log-command.py
  ```

  **PowerShell**:
  ```powershell
  '{}' | python hooks\scripts\log-command.py
  ```

- [ ] **Expected:** prints `{"continue": true}` and exits 0. No tracebacks, no `ModuleNotFoundError`, no `ImportError: cannot import name ...`.
- [ ] **Why this matters:** if the cclogger package can't be imported by the subprocess, every Claude Code hook invocation fails and no logging happens at all.

### HV.2 — Differential snapshot is byte-identical to v0.3.6 baseline

Validates: across the 25-event synthetic fixture (covering all hook events + tool categories + conversation events), the new modular code produces exactly the same log output as the pre-refactor monolith.

- [ ] **One-time setup** (skip if `tests/snapshots/v036_baseline/` already exists):
  ```
  git checkout v0.3.6 -- hooks/scripts/log-command.py
  python tests/snapshots/diff_check.py --capture-baseline
  git checkout HEAD -- hooks/scripts/log-command.py
  ```
- [ ] **Steps:**
  ```
  python tests/snapshots/diff_check.py
  ```
- [ ] **Expected:** `Byte-identical to baseline (8 log file(s))`. Exit code 0.
- [ ] **Why this matters:** this is the strongest mechanical verification the refactor preserved behavior. A red here is a regression somewhere in the dispatch chain.

### HV.3 — Plugin reload picks up the new file layout

Validates: the marketplace `directory` source can re-install from the modularized tree, and `~/.claude/plugins/cache/` ends up with the cclogger/ subfolder intact.

- [ ] **Steps**:
  1. `claude plugin update session-logger@dazzle-claude-plugins` (in any terminal where `claude` is on PATH)
  2. Inspect the cache (paths show the cmd.exe form; PowerShell uses `Get-ChildItem`, POSIX uses `ls`):

     **cmd.exe**:
     ```cmd
     dir "%USERPROFILE%\.claude\plugins\cache\dazzle-claude-plugins\session-logger" /s /b | findstr cclogger
     ```

     **PowerShell**:
     ```powershell
     Get-ChildItem "$env:USERPROFILE\.claude\plugins\cache\dazzle-claude-plugins\session-logger" -Recurse | Where-Object { $_.FullName -match 'cclogger' } | Select-Object FullName
     ```

     **POSIX**:
     ```bash
     find ~/.claude/plugins/cache/dazzle-claude-plugins/session-logger -path '*/cclogger/*'
     ```

- [ ] **Expected:** list of 14 files: `__init__.py` + 13 module files. If the count is wrong or `cclogger/` is missing entirely, the marketplace artifact dropped the new package.
- [ ] **Why this matters:** the test suite runs against the source tree, but Claude Code runs against the plugin cache. A green pytest with a broken cache means production is broken.

### HV.4 — Real session: tool entry has the rich Edit format preserved

Validates: in an actual live Claude Code session, the `Edit` handler still emits `{Edit: "path:line" ← "snippet" (-N/+ML)}` exactly as it did pre-refactor. This format is load-bearing for vim-jumpable + grep-survivable log readability per the user's design intent.

- [ ] **Steps:**
  1. Start a new Claude Code session in a throwaway directory:
     ```
     mkdir cclogger-checklist-hv4 && cd cclogger-checklist-hv4 && claude
     ```
  2. In the session, ask Claude to: `create a file foo.txt with three lines, then edit one line in the middle`.
  3. After Claude finishes, exit the session (`/exit`).
  4. Locate the sesslog directory:

     **PowerShell** (works on Windows):
     ```powershell
     Get-ChildItem "$env:USERPROFILE\.claude\sesslogs" -Directory | Where-Object { $_.Name -match 'cclogger-checklist-hv4' } | ForEach-Object { Get-ChildItem $_.FullName -Filter '*.tools_*.log' | Get-Content }
     ```

     **POSIX**:
     ```bash
     cat ~/.claude/sesslogs/*cclogger-checklist-hv4*/.tools_*.log
     ```

- [ ] **Expected:** at least one entry of the form:
  ```
  [[YYYY-MM-DD HH:MM:SS]] {Edit: "<full-path-to-foo.txt>:N" ← "<truncated-content>" (NL) }
  ```
  with the `:line-number`, the `←` arrow, and `(NL)` or `(-X/+YL)` line-delta suffix.
- [ ] **Why this matters:** if any of those pieces is missing (no line number, no arrow, no delta), the formatter dispatch broke during modularization and downstream grep workflows are silently degraded.

### HV.5 — Real session: convo channel captures user prompt + AI text

Validates: UserPromptSubmit, Stop, and SubagentStop hooks all run through `cclogger.conversation.handle_conversation_event` and write to `.convo_*.log`.

- [ ] **Steps:**
  1. Fresh session in a throwaway dir (`mkdir cclogger-checklist-hv5 && cd cclogger-checklist-hv5 && claude`)
  2. Type any prompt (e.g., `say hello in two words`)
  3. Wait for Claude's response
  4. Exit (`/exit`)
  5. View the convo log:

     **PowerShell**:
     ```powershell
     Get-ChildItem "$env:USERPROFILE\.claude\sesslogs" -Directory | Where-Object { $_.Name -match 'cclogger-checklist-hv5' } | ForEach-Object { Get-ChildItem $_.FullName -Filter '*.convo_*.log' | Get-Content }
     ```

     **POSIX**:
     ```bash
     cat ~/.claude/sesslogs/*cclogger-checklist-hv5*/.convo_*.log
     ```

- [ ] **Expected:** at least two entries, in order:
  ```
  [[ts]] {USER: "say hello in two words" }
  [[ts]] {AI: "Hello there" }
  ```
  (or whatever Claude actually said — the marker is `USER:` for the prompt and `AI:` for the response).
- [ ] **Why this matters:** the conversation handler integrates with the SessionLogger constructor, the cursor file (`~/.claude/session-states/<id>.convo-cursor`), and the transcript JSONL reader — any one of which could break in subtle ways post-refactor.

### HV.6 — Session resume after restart writes a fresh SESSION-START marker

Validates: the `~/.claude/session-states/<id>.started` flag clearing logic (only on `SessionStart` hook event) and `get_run_number` correctly increment Run #N across resumes.

- [ ] **Steps:**
  1. From a fresh session in a throwaway dir, run any command (e.g., ask Claude to `list files in this dir`).
  2. Exit (`/exit`).
  3. Re-enter the same session: `claude --resume` (pick the session you just created from the menu).
  4. Run another command in the resumed session.
  5. Exit and view the sesslog file:

     **POSIX example** (adapt for PowerShell using `Get-Content -Tail`):
     ```bash
     cat ~/.claude/sesslogs/*cclogger-checklist-hv6*/.sesslog_*.log | grep -A1 "SESSION START"
     ```

- [ ] **Expected:** TWO `SESSION START` marker blocks (rule lines + `═══ SESSION START • ts • Run #1 • name`, then later `Run #2`). Compaction count separate; if you triggered `/compact`, you'd see `CONTEXT COMPACTED • Compaction #1` instead of a second SESSION START.
- [ ] **Why this matters:** the run-counter logic spans `cclogger/markers.py` (counting) + `cclogger/logger.py` (writing) + `log-command.py` main (clearing the `.started` flag). Any miswiring after modularization shows up as missing or duplicated markers.

## Section 2: Live tool events — verify rich format preservation

### 2.1 Bash command logged with full text

- [ ] **Steps:** in a fresh throwaway session, run `Bash` (e.g., ask Claude to run `dir` or `ls -la`). Then `cat` the `.shell_*.log`.
- [ ] **Expected:** entry of form `[[ts]] {Bash: dir }` (verbosity 2 default). Command preserved literally.

### 2.2 Edit handler emits `path:line ← snippet (delta)` (HV.4 deep-dive)

- [ ] **Steps:** edit a file with a meaningful change (more than one line modified). View `.tools_*.log`.
- [ ] **Expected:** `{Edit: "path:N" ← "preview..." (-2/+3L) }`. The `:line` is found via `find_line_number` (formatters.py); the `(-N/+ML)` delta comes from comparing line counts of old_string vs new_string.

### 2.3 Write handler emits `path ← preview (NL)`

- [ ] **Steps:** create a new file via Write. View `.tools_*.log`.
- [ ] **Expected:** `{Write: "<path>" ← "<first chars of content>..." (NL) }` where `N` is the line count.

### 2.4 Grep handler emits `pattern in path/`

- [ ] **Steps:** ask Claude to grep for a pattern in a directory.
- [ ] **Expected:** `{Grep: <pattern> in "<resolved-path>/" }` (note the trailing slash for directories) or `{Grep: <pattern> | "<file>" }` (for single-file paths).
- [ ] **Why this matters:** the resolved path is platform-dependent (`Path.resolve()`); the format must survive both Windows backslashes and POSIX forward slashes.

### 2.5 Skill invocation logged with truncated args preview

- [ ] **Steps:** invoke a skill (e.g., `/test-checklist`). View `.tools_*.log`.
- [ ] **Expected:** `{Skill: <skill-name> ← "<truncated-args>" }` (truncation per `performance.skill_args_length`, default 100 chars).

### 2.6 TodoWrite logged as JSON

- [ ] **Steps:** trigger a TodoWrite. View `.tools_*.log`.
- [ ] **Expected:** `{TodoWrite: [{...},...]}` — compact JSON with `separators=(",", ":")`.

### 2.7 Unknown tool gets `?` prefix + routes to `.unknowns_*.log`

- [ ] **Steps:** invoke a tool not in `TOOL_CATEGORIES` (e.g., a recently-added one). Or simulate by piping crafted JSON into the hook directly:

  ```
  echo {"tool_name":"FrobnicateBlobs","tool_input":{"foo":"bar"},"hook_event_name":"PostToolUse","session_id":"checklist-2-7","transcript_path":""} | python hooks/scripts/log-command.py
  ```

- [ ] **Expected:** entry in `.unknowns_*.log` (and `.sesslog_*.log`) of form `{?FrobnicateBlobs: ... }`. NOT in `.shell_*.log` or `.tools_*.log`. AND a single line in `~/.claude/logs/hook-debug.log` warning about the unknown tool (sentinel created at `~/.claude/logs/.unknown_tool_warnings/FrobnicateBlobs.warned`).
- [ ] **Reset:** delete the sentinel between runs if you want to re-trigger the warning:

  **POSIX**:
  ```bash
  rm -f ~/.claude/logs/.unknown_tool_warnings/FrobnicateBlobs.warned
  ```

  **PowerShell**:
  ```powershell
  Remove-Item "$env:USERPROFILE\.claude\logs\.unknown_tool_warnings\FrobnicateBlobs.warned" -ErrorAction SilentlyContinue
  ```

## What the automated tests DON'T cover

These are the things ONLY this checklist verifies — written here as the explicit gap so future maintainers know what's manual-only:

- **Real subprocess invocation by Claude Code's Node entry point** (`run-hook.mjs`). Tests pipe JSON into `python log-command.py` directly; they don't exercise the `node → python` shell-out path or the marketplace cache.
- **Plugin install/cache layout from the marketplace artifact**. The `directory` source is supposed to surface every file in the source tree under `~/.claude/plugins/cache/`, but cache invalidation, file-mtime preservation, and missing-file detection are all marketplace-side and unmocked.
- **dazzle_filekit auto-install in a real venv with no internet**. Mocked tests assume it's already importable; the escalating-pip-strategies path is only exercised manually.
- **Live transcript JSONL schema**. The tests use a synthetic transcript with shapes WE control. Claude Code's actual transcript has `{"type": "assistant", "message": {"content": [...]}}`, but the schema isn't formally pinned and could evolve with Claude Code updates.
- **Windows codepage (cp437/cp1252) interaction with subprocess stdin/stdout**. Tests run with whatever codepage your terminal happens to have; checklist sections 8.1 explicitly verifies UTF-8 decode under cp437/cp1252.
- **Symlink creation fallback chain on Windows**. `os.symlink → dazzlelink → mklink` per dazzle_filekit. Without admin rights or developer mode, only the `mklink` path works — automated tests don't reproduce the permission failure modes.
- **Path.resolve() platform-dependence**. The diff_check baseline is captured per-machine for exactly this reason — tests can't verify cross-platform identical output, only same-machine identical output.
- **Plugin reload race conditions**. If a session is open while you `claude plugin update`, the running session's hook subprocesses may resolve differently than the cache state implies.
- **/rename via the actual Claude Code rename UI** (vs. directly writing `customTitle` to the transcript). The integration with the rename UI is verified end-to-end only here.
- **`/compact` actually firing the SessionStart-with-source=compact hook event**. The tests synthesize that event; only a real session under `/compact` confirms Claude Code emits it.
- **Performance feel**. Hook subprocess startup time is sensitive to Python interpreter cold-start, dazzle_filekit import cost, and config-file reads. A "feels slow" reading from a human is hard to mock.
- **Error message readability**. `~/.claude/logs/hook-debug.log` lines should make sense to a human grepping them six months later — humans verify, mocks don't.
