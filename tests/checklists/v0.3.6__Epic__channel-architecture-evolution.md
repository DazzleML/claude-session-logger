## What this is

End-to-end verification of the v0.3.x channel architecture evolution epic. Most behavior in this release depends on **live Claude Code event payloads and transcript schemas** that aren't formally pinned and that mocks can't fully exercise. This checklist is the gate before we declare these features stable. Specifically:

- New `tools` and `convo` channels appear under `~/.claude/sesslogs/`
- Conversation hooks (`UserPromptSubmit`, `Stop`, `SubagentStop`) actually capture content from live Claude Code events
- Per-channel config directory layout coexists with single-file layout
- Subtype routing produces correctly-named per-subtype log files when opted in

## Prerequisites

1. **Plugin v0.3.6 installed** via `claude plugin install session-logger@dazzle-claude-plugins` or local marketplace
2. **A test Claude Code session** in any project (or accept that you'll have churn in your live sesslogs)
3. **Backup your real config** if you have one at `~/.claude/plugins/settings/session-logger.json` — Section 3 tests overwrite it

### Reset between sections

Most tests are non-destructive. For tests that depend on a clean throttle/cursor state:

- **Throttle reset** (for unknown-tool warnings, rare):
  - **PowerShell**: `Remove-Item -Recurse -Force "$env:USERPROFILE\.claude\logs\.unknown_tool_warnings" -ErrorAction SilentlyContinue`
  - **POSIX**: `rm -rf ~/.claude/logs/.unknown_tool_warnings`
  - **cmd.exe**: `if exist "%USERPROFILE%\.claude\logs\.unknown_tool_warnings" rmdir /s /q "%USERPROFILE%\.claude\logs\.unknown_tool_warnings"`

- **Convo cursor reset** (for re-reading transcripts from start):
  - **PowerShell**: `Get-ChildItem "$env:USERPROFILE\.claude\session-states\*.convo-cursor" | Remove-Item -ErrorAction SilentlyContinue`
  - **POSIX**: `rm -f ~/.claude/session-states/*.convo-cursor`
  - **cmd.exe**: `del "%USERPROFILE%\.claude\session-states\*.convo-cursor" 2>nul`

### State isolation note

These tests interact with your live Claude Code session state. To run with full isolation, point Claude Code at a separate `CLAUDE_HOME` (advanced — not covered here). Most testers can just accept the test-session churn and `dz safedel` the test session dirs at the end.

### IMPORTANT: Conversation hooks affect ALL active Claude Code sessions

When v0.3.6 is installed, the new `UserPromptSubmit`, `Stop`, and `SubagentStop` hooks fire on every turn in every active Claude Code session on the machine — not just your test session. If something is wrong (e.g., the `UserPromptSubmit` payload field name doesn't match what we expect), you'll see empty `{USER:  }` entries or errors in `~/.claude/logs/hook-debug.log` for your real work too.

**Recommended test isolation** (combine A+B for maximum safety):

**A. Test in a separate Claude Code instance**: open a NEW Claude Code window in a throwaway directory (e.g., `mkdir ~/cclogger-test && cd ~/cclogger-test && claude`). Don't test in a window where you have important real work in progress.

**B. Configure minimal-footprint test config** before installing v0.3.6: enable only the `convo` channel during testing so that even if something is buggy, only one log file is affected per session.

  **PowerShell**:
  ```powershell
  $cfg = "$env:USERPROFILE\.claude\plugins\settings\session-logger.json"
  '{
    "routing": {
      "channels": {
        "shell":    {"file_prefix": ".shell_",    "enabled": false},
        "sesslog":  {"file_prefix": ".sesslog_",  "enabled": false},
        "tasks":    {"file_prefix": ".tasks_",    "enabled": false},
        "tools":    {"file_prefix": ".tools_",    "enabled": false},
        "convo":    {"file_prefix": ".convo_",    "enabled": true},
        "unknowns": {"file_prefix": ".unknowns_", "enabled": false}
      }
    }
  }' | Set-Content $cfg -Encoding UTF8
  ```

  **POSIX**:
  ```bash
  cat > ~/.claude/plugins/settings/session-logger.json <<'EOF'
  {
    "routing": {
      "channels": {
        "shell":    {"file_prefix": ".shell_",    "enabled": false},
        "sesslog":  {"file_prefix": ".sesslog_",  "enabled": false},
        "tasks":    {"file_prefix": ".tasks_",    "enabled": false},
        "tools":    {"file_prefix": ".tools_",    "enabled": false},
        "convo":    {"file_prefix": ".convo_",    "enabled": true},
        "unknowns": {"file_prefix": ".unknowns_", "enabled": false}
      }
    }
  }
  EOF
  ```

After verifying the conversation channel works, you can re-enable other channels and re-run HV.1 / HV.4 / HV.5 / HV.6 with the full default set.

**C. Quick rollback path** if something breaks badly: revert to v0.2.1 cached version with `claude plugin install session-logger@dazzle-claude-plugins --version 0.2.1`. That removes the new hooks entirely.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact behaviors v0.3.6 introduces. If any fail, the epic is broken.

### HV.1 — `.tools_*.log` file is created and captures AI activity

Validates: new `tools` channel routing (#28).

- [ ] **Steps**: Start a new Claude Code session. Run several tool operations (e.g., `Bash`, `Read`, `Edit`).

  Then check:

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.tools_*"
  ```

  **POSIX**:
  ```bash
  ls ~/.claude/sesslogs/*/.tools_*
  ```

  **cmd.exe**:
  ```cmd
  dir /s /b "%USERPROFILE%\.claude\sesslogs\.tools_*"
  ```

- [ ] **Expected**: A `.tools_<context>.log` file exists in the active session directory, containing the tool calls you triggered (Bash, Read, Edit lines).
- [ ] **Why this matters**: The user's primary "find exact tool calls" investigation channel. If absent, the entire v0.3.0 work didn't ship.

### HV.2 — `.convo_*.log` captures user prompts via `UserPromptSubmit`

Validates: conversation channel + UserPromptSubmit handler (#33).

- [ ] **Steps**: In an active Claude Code session, type a distinct test prompt: `"Test prompt for v0.3.6 verification - this should appear in .convo_*"` and submit. Then:

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.convo_*" |
    Sort-Object LastWriteTime -Descending |
    Select-Object -First 1 |
    Get-Content -Tail 5
  ```

  **POSIX**:
  ```bash
  ls -t ~/.claude/sesslogs/*/.convo_* | head -1 | xargs tail -5
  ```

- [ ] **Expected**: A line like `[[YYYY-MM-DD HH:MM:SS]] {USER: "Test prompt for v0.3.6 verifi..." }` (truncated to 200 chars by default).
- [ ] **Why this matters**: If empty or absent, the `UserPromptSubmit` handler doesn't recognize the actual payload field — handler tries `user_prompt`, `prompt`, `user_input` in order; need to verify which one Claude Code actually emits.

### HV.3 — `.convo_*.log` captures AI responses via `Stop`

Validates: Stop hook + transcript-read for AI capture (#34).

- [ ] **Steps**: After Claude finishes responding to your HV.2 prompt, check:

  **PowerShell**:
  ```powershell
  Get-Content (Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.convo_*" | Sort-Object LastWriteTime -Descending | Select-Object -First 1) -Tail 10
  ```

  **POSIX**:
  ```bash
  ls -t ~/.claude/sesslogs/*/.convo_* | head -1 | xargs tail -10
  ```

- [ ] **Expected**: One or more lines like `[[YYYY-MM-DD HH:MM:SS]] {AI: "<Claude's response..." }` after the USER line.
- [ ] **Why this matters**: This depends on the Stop event firing AFTER the transcript is fully flushed AND the transcript schema matching what `_extract_text_from_assistant_entry` expects (handles both `message.content` and `role/content` shapes). If empty, schema may have changed or there's a timing race.

### HV.4 — `.shell_*.log` includes Grep/LS/Glob entries (workflow context audit)

Validates: bash category audit (#29).

- [ ] **Steps**: In an active Claude Code session, ask Claude to use `Grep`, `LS`, and `Glob`. Then:

  **PowerShell**:
  ```powershell
  Get-Content (Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.shell_*" | Sort-Object LastWriteTime -Descending | Select-Object -First 1) -Tail 30 | Select-String -Pattern '\{(Grep|LS|Glob):'
  ```

  **POSIX**:
  ```bash
  ls -t ~/.claude/sesslogs/*/.shell_* | head -1 | xargs tail -30 | grep -E '\{(Grep|LS|Glob):'
  ```

- [ ] **Expected**: Lines like `{Grep: pattern}`, `{LS: "/path"}`, `{Glob: "*.py"}` present in `.shell_*.log` (alongside `Bash` and `PowerShell` entries).
- [ ] **Why this matters**: The audit decision was that these belong with shell-equivalent ops in the curated history channel. If they only appear in `.tools_*` and `.sesslog_*` (not `.shell_*`), the categorization didn't take effect.

### HV.5 — Per-channel config directory layout works

Validates: ConfigLoader directory-layout support (#30).

- [ ] **Steps**:
  1. Create the directory layout:

     **PowerShell**:
     ```powershell
     $cfg = "$env:USERPROFILE\.claude\plugins\settings\session-logger"
     New-Item -ItemType Directory -Force -Path "$cfg\channels"
     '{ "performance": { "skill_args_length": 50 } }' | Set-Content "$cfg\_global.json" -Encoding UTF8
     '{ "file_prefix": ".convo_", "enabled": false }' | Set-Content "$cfg\channels\convo.json" -Encoding UTF8
     ```

     **POSIX**:
     ```bash
     mkdir -p ~/.claude/plugins/settings/session-logger/channels
     echo '{ "performance": { "skill_args_length": 50 } }' > ~/.claude/plugins/settings/session-logger/_global.json
     echo '{ "file_prefix": ".convo_", "enabled": false }' > ~/.claude/plugins/settings/session-logger/channels/convo.json
     ```

  2. Start a new Claude Code session and trigger any tool. Type a user prompt.

- [ ] **Expected**:
  - Hook still works (no errors in `~/.claude/logs/hook-debug.log`)
  - NO `.convo_*.log` file is created (channel disabled via the per-channel file)
  - Skill args truncated to 50 chars (verify by triggering a `/skillname args` invocation)
- [ ] **Why this matters**: The directory layout is a substantial refactor of how config is loaded; if it doesn't work, users can't adopt the discoverable layout.

### HV.6 — Subtype routing produces per-subtype channel files

Validates: subtype routing framework (#31).

- [ ] **Steps**:
  1. Add subtype routing to your config:

     **PowerShell** (single-file layout):
     ```powershell
     '{ "routing": { "subtype_routing": { "bash": true } } }' | Set-Content "$env:USERPROFILE\.claude\plugins\settings\session-logger.json" -Encoding UTF8
     ```

     **POSIX**:
     ```bash
     echo '{ "routing": { "subtype_routing": { "bash": true } } }' > ~/.claude/plugins/settings/session-logger.json
     ```

     **NOTE**: If you did HV.5, remove the directory first or it will win:

     **PowerShell**: `Remove-Item -Recurse -Force "$env:USERPROFILE\.claude\plugins\settings\session-logger"`

     **POSIX**: `rm -rf ~/.claude/plugins/settings/session-logger`

  2. Start a new Claude Code session, run `Bash` and (on Windows) `PowerShell` calls.

- [ ] **Expected**:
  - `.shell-bash_<context>.log` exists with Bash entries only
  - `.shell-powershell_<context>.log` (Windows) exists with PowerShell entries only
  - `.tools-bash_*` and `.tools-powershell_*` (Windows) similarly split
  - Original `.shell_*.log` still receives all entries (subtype channels are ADDITIVE)
- [ ] **Why this matters**: Subtype routing is the most powerful new feature for power users; it generalizes to MCP servers, agents, skills.

## What the automated tests DON'T cover

These are the things this checklist exists to verify:

- **Real `UserPromptSubmit` event payload** — handler tries 3 field names; only live verification confirms which one Claude Code actually emits
- **Real `Stop` and `SubagentStop` event timing** — assumption: transcript is flushed before Stop fires. Race condition possible if not.
- **Real transcript JSONL schema** — handler tries 2 shapes (`message.content` and `role/content`). New SDK versions could break this.
- **File appearance under live `~/.claude/sesslogs/`** — tests use temp dirs; the actual subprocess hook execution + file creation is unverified
- **Multi-turn conversation duplicate prevention** — cursor logic tested in isolation; real multi-event sequences need verification
- **Subagent text capture from real Task invocations** — only verifiable by spawning real subagents
- **MCP subtype extraction with real MCP server names** — tested with mock data
- **Cross-shell command behavior in this checklist** — checklist commands tested by you running them
- **Performance impact of 5 simultaneous channel writes per event** — measurable cost; presumably negligible but unmeasured
- **Schema IDE autocompletion** — verified via VS Code visual confirmation only
