# Human Test Checklist: scripts-repo subtree migration + extra-targets sync-versions

**Target version:** v0.2.0
**Ships with commits:** parent #20 (sub-issues #24 + #25 + #26)
**Companion automated tests:** none new yet for this work; existing `tests/one-offs/test_rename_corruption.py` (15 tests, unrelated to this work) remain green

## What this is

This document verifies the end-to-end behavior of the migration from a hand-maintained `scripts-repo/` to a `git-repokit-common` subtree, AND the local generalization of the upstream `sync-versions.py` to handle plugin-specific JSON version files via a new `extra-targets` config.

There are very few automated tests for this work because most of the surface area is filesystem layout, subtree mechanics, gitignore behavior, and subprocess invocation through git hooks — exactly the things automation systematically misses. **A human walks this list before declaring v0.2.0 ready to tag.**

## Prerequisites

1. **OS / shell**: Windows 10/11 with Git Bash (MINGW64). Most commands are also given for cmd.exe and PowerShell.
2. **Python**: `python --version` reports 3.9+. Verify with `python --version`.
3. **`git` and `gh` CLIs**: present and authenticated (`gh auth status` should report logged in).
4. **Working tree clean** (or at most has the staged commit-1 changes from #25). Verify with `git status`.
5. **Backup tag** `pre-subtree-migration-v0.1.11` exists (created earlier in #24). Verify: `git tag | grep pre-subtree`.
6. **No global state contamination**: this checklist deliberately operates on the real plugin cache when testing `dev-refresh.py`. Section 4 documents how to safely reset.

### State isolation

There is no test-isolation env var for sync-versions.py at this time -- it always reads the project's `pyproject.toml` and `version.py`. To protect against accidental version bumps during the checklist:

- Any test that runs a real `--bump` should be done on a throwaway branch (`git checkout -b checklist-test-NNN`).
- Reset between sections: `git checkout main && git branch -D checklist-test-NNN && git restore version.py .claude-plugin/`.

## Section 7: Pre-commit hook regressions (existing functionality)

These verify that our changes don't break the existing pre-commit protections.

### Section 7 PRE-FLIGHT — stash any staged work first

**Critical safety step** (added after a 2026-04-20 incident where staged Commit 2a content was lost during Section 7.1). The mutation tests in this section use `git checkout -b ... + git commit + git branch -D`. If the index has staged-but-not-yet-committed work, that staged content gets folded into the throwaway-branch commit and is lost when the branch is deleted.

- [ ] **Before any test in Section 7, stash any staged or untracked work**:

  **Git Bash**:
  ```bash
  git stash --include-untracked  # harmless no-op if there's nothing staged
  ```

  **At the END of Section 7** (after all sub-tests):
  ```bash
  git stash list  # confirm the stash is there
  git stash pop   # restore the staged work
  ```

  Alternatively: skip Section 7 entirely if you have staged work for an upcoming commit. Recovery if you skip the stash and lose data: `git reflog | head -20` will show the unreachable commit; `git checkout <sha> -- <path>` recovers individual files.

### 7.1 Private content protection still works

- [ ] **Steps** (Git Bash):

## Section 8: Differential equivalence with pre-subtree behavior

This section uses `scripts-repo/local/diff-harness.py` to prove that the NEW sync-versions.py implementation produces the same SemVer outputs as the OLD pre-subtree-migration version. The harness clones the repo into `%TEMP%` (NOT git worktrees -- worktrees share `.git/` and would leak commits/hooks), checks out the OLD ref in one copy and HEAD in the other, runs identical version operations in both, and diffs the results.

### Safety constraints (must hold)

- [ ] **Test copies live in `%TEMP%`** via `git clone --no-local` so `.git/` is fully independent
- [ ] **Pre-commit / post-commit / pre-push hooks deleted** in each copy before any operation
- [ ] **No `--dev-refresh` against real plugin versions**: only bogus versions like `99.99.99` may be used to validate path-construction
- [ ] **Mandatory cleanup**: temp copies removed (unless `--keep` specified for debugging)
- [ ] **Leakage check at end**: confirm `~/.claude/plugins/cache/dazzle-claude-plugins/session-logger/` and the original repo's `git log --oneline -3` are unchanged after the harness runs

The harness automates all of these checks. The leakage check FAILS the run if anything outside the test copies was mutated.

### 8.1 Run the harness against current state

- [ ] **Steps** (Git Bash / cmd / PowerShell -- harness is portable Python):
  ```
  python scripts-repo/local/diff-harness.py
  ```
- [ ] **Expected** (BEFORE Commit 2b ships -- documents the gap):
  - Cleans up `%TEMP%` copies at the end
  - "[LEAKAGE CHECK] [OK] No leakage detected"
  - SemVer comparison shows `version.py components` MATCH (`0.1.12 == 0.1.12`)
  - SemVer comparison shows `plugin.json` and `marketplace.json` DIFFER (OLD: `0.1.12`, NEW: `0.1.11`) -- this is the spec for what extra-targets must implement
  - Exit code 1 (FAIL is expected and correct at this stage)

- [ ] **Expected** (AFTER Commit 2b ships -- proves equivalence):
  - SemVer comparison shows ALL three sources MATCH
  - Normalized file diff is empty (or only contains expected stylistic differences like the `PHASE = None` vs `PHASE = ""` upstream convention)
  - Exit code 0

### 8.2 Inspect a failing run

- [ ] **Steps**:
  ```

  ```
- [ ] **Expected**:
  - `--keep` leaves the temp copies; the path is printed at the end ("CLEANUP SKIPPED")

## What the automated tests DON'T cover (the whole point of this checklist)

- **gitignore quirks**: anchored vs unanchored rules, parent-exclusion sticky behavior, Windows `\` vs `/` paths
- **Pre-commit hook subprocess loops**: a misbehaving hook can fork-bomb itself; only a real commit detects this
- **Subtree pull merges**: simulating a real `git subtree pull` against upstream is hard to mock
- **Plugin cache filesystem behavior**: real `~/.claude/plugins/cache/` interaction
- **Cross-shell command parsing**: what works in Git Bash may behave differently in cmd.exe
- **Real Claude Code plugin reinstall flow**: tests can't actually run `claude plugin install`
- **Pre-commit interactions with version.py auto-update**: could it accidentally bump the version twice?
- **Backward-compat for >10 other consumers**: only real-world testing on those projects can fully verify
