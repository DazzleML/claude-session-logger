**Ships with commit:** TBD (pending sign-off; CHANGELOG entry already drafted)

## What this is

End-to-end verification that v0.2.1 actually behaves correctly inside a live Claude Code session — covering things mocks can't: real subprocess hook invocation, file appearance under `~/.claude/sesslogs/`, cross-shell PowerShell vs Bash detection, and the cross-process throttling that the per-process unit test can't fully prove.

This document is for project maintainers and external testers verifying the v0.2.1 release. The 5-minute "High-Value Verification" section is the smoke test; sections below cover edge cases, error handling, and integration.

## Prerequisites

1. **Windows** with Python 3.9+ and Claude Code installed (this checklist exercises both Bash and PowerShell tools — runs on Linux/macOS too but PowerShell sections may be skipped or done with `pwsh`)
2. **Plugin v0.2.1 installed** via `claude plugin install session-logger@dazzle-claude-plugins`
3. **Reset throttling** before running discovery tests — see "Reset between sections"

### Reset between sections

The sentinel-file throttle is **persistent across sessions and processes**. To re-trigger an "unknown tool encountered" warning, delete the sentinels:

**cmd.exe**:
```cmd
if exist "%USERPROFILE%\.claude\logs\.unknown_tool_warnings" rmdir /s /q "%USERPROFILE%\.claude\logs\.unknown_tool_warnings"
```

**PowerShell**:
```powershell
$dir = "$env:USERPROFILE\.claude\logs\.unknown_tool_warnings"
if (Test-Path $dir) { Remove-Item -Recurse -Force $dir }
```

**POSIX**:
```bash
rm -rf ~/.claude/logs/.unknown_tool_warnings
```

### State isolation

This checklist does NOT require config isolation — it reads the live `~/.claude/sesslogs/` and `~/.claude/logs/` to verify real hook behavior. If you want to run without affecting your live data, point Claude Code at a separate `CLAUDE_HOME` (advanced — not covered here).

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact behaviors v0.2.1 introduces. If any fail, the work is broken.

### HV.1 — PowerShell tool logs the actual command (not empty)

Validates: PowerShell handler addition. The reason for the v0.2.1 release.

- [ ] **Steps** (run inside a Claude Code session on Windows):

  Ask Claude Code to run a PowerShell command, e.g.:
  > "Run a PowerShell command to show the current directory: `Get-Location`"

  Then inspect the most recent sesslog:

  **cmd.exe**:
  ```cmd
  for /f "delims=" %%i in ('dir /b /od /a-d "%USERPROFILE%\.claude\sesslogs\*\.shell_*.log"') do set LATEST=%%i
  type "%USERPROFILE%\.claude\sesslogs\<your-session-dir>\.shell_<context>.log" | findstr PowerShell
  ```

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.shell_*.log" |
    Sort-Object LastWriteTime -Descending |
    Select-Object -First 1 |
    Get-Content -Tail 20 |
    Select-String "PowerShell"
  ```

  **POSIX**:
  ```bash
  ls -t ~/.claude/sesslogs/*/.shell_*.log | head -1 | xargs tail -20 | grep PowerShell
  ```

- [ ] **Expected:** Recent log line shows the command, e.g.:
  ```
  [[2026-05-01 14:15:30]] {PowerShell: Get-Location }
  ```
  NOT `{PowerShell:  }` (the bug being fixed).

- [ ] **Why this matters:** The whole reason for the v0.2.1 release. If this is empty, the release didn't ship correctly.

### HV.2 — Dedicated `.unknowns_*.log` channel exists for unknown tools

Validates: New `unknowns` channel in defaults, correct routing.

- [ ] **Steps**:

  In a Claude Code session, trigger any tool Claude Code added that this plugin version doesn't know about. **In practice during testing, you may need to invoke a tool that's not in `TOOL_CATEGORIES`** — examples that may be uncategorized depending on Claude Code version: `Monitor`, `RemoteTrigger`, `LSP`, `EnterWorktree`, `ExitWorktree`. If none are available, see Section 1.3 for a synthetic injection method.

  After the tool runs, check for the new channel file:

  **cmd.exe**:
  ```cmd
  dir /b "%USERPROFILE%\.claude\sesslogs\<your-session-dir>\.unknowns_*"
  ```

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.unknowns_*"
  ```

  **POSIX**:
  ```bash
  ls ~/.claude/sesslogs/*/.unknowns_*
  ```

- [ ] **Expected:** A `.unknowns_<context>.log` file exists in the session directory, containing the unknown tool's entry.

- [ ] **Why this matters:** Verifies the new channel is created and routes correctly. Without this, unknown tools have no dedicated discovery surface.

### HV.3 — `?` marker appears for unknown tools

Validates: Marker prefix in `generate_entry()`.

- [ ] **Steps**: After triggering an unknown tool (HV.2), grep for the `?` marker pattern in the sesslog:

  **cmd.exe**:
  ```cmd
  findstr "{?" "%USERPROFILE%\.claude\sesslogs\<your-session-dir>\.sesslog_*.log"
  ```

  **PowerShell**:
  ```powershell
  Select-String -Pattern '\{\?' -Path "$env:USERPROFILE\.claude\sesslogs\*\.sesslog_*.log"
  ```

  **POSIX**:
  ```bash
  grep '{?' ~/.claude/sesslogs/*/.sesslog_*.log
  ```

- [ ] **Expected:** Lines like:
  ```
  [[2026-05-01 14:20:00]] {?Monitor: process-id-12345 }
  ```
  The `?` is INSIDE the brace, immediately before the tool name.

- [ ] **Why this matters:** Users who don't know about the dedicated `.unknowns_*` channel can still grep their sesslog for new tools.

### HV.4 — Shell channel stays clean (no unknown-tool pollution)

Validates: `unknown` route deliberately excludes `shell` channel.

- [ ] **Steps**: Confirm the unknown tool from HV.2/HV.3 does NOT appear in `.shell_*.log`:

  **cmd.exe**:
  ```cmd
  findstr /C:"<unknown-tool-name>" "%USERPROFILE%\.claude\sesslogs\<your-session-dir>\.shell_*.log"
  ```

  **PowerShell**:
  ```powershell
  Select-String -Pattern "<unknown-tool-name>" -Path "$env:USERPROFILE\.claude\sesslogs\*\.shell_*.log"
  ```

  **POSIX**:
  ```bash
  grep "<unknown-tool-name>" ~/.claude/sesslogs/*/.shell_*.log
  ```

- [ ] **Expected:** No matches. (`.shell_*.log` should contain only Bash, PowerShell, and other `bash`-category tool entries.)

- [ ] **Why this matters:** Channel philosophy — `.shell_*.log` should be a clean, copy-pasteable shell history. Pre-v0.2.1, unknown tools polluted it.

### HV.5 — Throttled warning fires exactly once in hook-debug.log

Validates: Cross-process sentinel-file throttling.

- [ ] **Steps**:
  1. Reset throttle (see "Reset between sections" above)
  2. Trigger an unknown tool 3+ times in a Claude Code session (e.g., ask Claude to run an unknown tool repeatedly, or invoke the same `LSP`/`Monitor` action multiple times)
  3. Inspect hook-debug.log:

  **cmd.exe**:
  ```cmd
  findstr "Unknown tool" "%USERPROFILE%\.claude\logs\hook-debug.log"
  ```

  **PowerShell**:
  ```powershell
  Select-String -Pattern "Unknown tool '<TOOL>'" -Path "$env:USERPROFILE\.claude\logs\hook-debug.log"
  ```

  **POSIX**:
  ```bash
  grep "Unknown tool" ~/.claude/logs/hook-debug.log
  ```

- [ ] **Expected:** Exactly ONE `Unknown tool '<ToolName>' encountered with no extractable content - input fields: [...]` line per distinct tool, regardless of how many times the tool was invoked.

- [ ] **Check sentinel exists:**

  **cmd.exe**:
  ```cmd
  dir "%USERPROFILE%\.claude\logs\.unknown_tool_warnings\"
  ```

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\logs\.unknown_tool_warnings\"
  ```

  **POSIX**:
  ```bash
  ls ~/.claude/logs/.unknown_tool_warnings/
  ```

  Should show `<ToolName>.warned` (one file per warned tool).

- [ ] **Why this matters:** The previous per-process `set` was effectively broken (each hook subprocess starts fresh). Sentinel files are the correctness fix that makes throttling actually work.

### HV.6 — Customized config still gets new defaults via merge

Validates: ConfigLoader's existing per-key merge behavior covers the new entries.

- [ ] **Steps**: Create or open `~/.claude/plugins/settings/session-logger.json` and add a custom channel WITHOUT mentioning `unknowns`:

  ```json
  {
    "$schema": "../../../../.claude/plugins/cache/dazzle-claude-plugins/session-logger/0.2.1/hooks/schemas/session-logger.schema.json",
    "routing": {
      "channels": {
        "shell": {"file_prefix": ".shell_", "enabled": true},
        "sesslog": {"file_prefix": ".sesslog_", "enabled": true},
        "tasks": {"file_prefix": ".tasks_", "enabled": true},
        "myCustomChannel": {"file_prefix": ".custom_", "enabled": false}
      },
      "category_routes": {
        "_default": ["shell", "sesslog"],
        "task": ["shell", "sesslog", "tasks"]
      }
    }
  }
  ```

  Trigger an unknown tool. Verify behavior matches a fresh install (HV.2 + HV.4 still pass: dedicated channel exists, shell stays clean).

- [ ] **Expected:** Despite no explicit `unknowns` channel or `unknown` route in user config, the merge fills in defaults. `.unknowns_*.log` appears, `.shell_*.log` stays unpolluted.

- [ ] **Why this matters:** Backwards compatibility for users who customized `category_routes`. Without merge behavior, customized users would silently miss the new improvements.

## Section 1: PowerShell handler — detailed coverage

### 1.1 PowerShell appears in `.shell_*.log` (correct routing)

- [ ] **Steps**: Run a PowerShell command in Claude Code, then check shell channel:

  **PowerShell**:
  ```powershell
  Select-String -Pattern "PowerShell" -Path "$env:USERPROFILE\.claude\sesslogs\*\.shell_*.log" |
    Select-Object -Last 5
  ```

- [ ] **Expected:** PowerShell entries appear in `.shell_*.log` (correct — PowerShell is `bash` category).

### 1.2 PowerShell description does NOT replace the command

- [ ] **Steps**: Trigger a PowerShell call with `description` field set (Claude Code typically sets this for context). Verify the log shows the `command` field, not `description`.
- [ ] **Expected:** `{PowerShell: <actual command>}`, NOT `{PowerShell: <description text>}`.

### 1.3 Synthetic unknown tool injection (when no real unknown is available)

If your Claude Code build doesn't expose any uncategorized tools, you can synthesize one for testing:

**POSIX/PowerShell** (run from project root):
```bash
echo '{"hook_event_name":"PostToolUse","tool_name":"FakeTestTool","tool_input":{"command":"hello"},"session_id":"test-fake-12345","transcript_path":"/tmp/no-such-transcript.jsonl","cwd":"/tmp"}' | python hooks/scripts/log-command.py
```

```powershell
'{"hook_event_name":"PostToolUse","tool_name":"FakeTestTool","tool_input":{"command":"hello"},"session_id":"test-fake-12345","transcript_path":"C:\\nope.jsonl","cwd":"C:\\"}' | python hooks\scripts\log-command.py
```

This invokes the hook with a fabricated unknown tool. Then check sesslogs for `{?FakeTestTool: hello }`.

> **Note**: This will create a fake session directory under `~/.claude/sesslogs/`. Clean up with `rm -rf ~/.claude/sesslogs/*test-fake-12345*` (POSIX) or equivalent.

- [ ] **Steps**: In `~/.claude/plugins/settings/session-logger.json`:

  Trigger an unknown tool.
- [ ] **Expected:** No `.unknowns_*.log` file is written for that session. (Entry still appears in `.sesslog_*.log` since the route includes both.)

- [ ] **Steps**: In config:
  ```json
  { "routing": { "category_routes": { "unknown": ["sesslog"] } } }
  ```
  Trigger an unknown tool.
- [ ] **Expected:** Entry appears only in `.sesslog_*.log`. No `.unknowns_*.log` written.

- [ ] **Steps**: In config:

## Section 4: Edge cases

### 4.1 Tool name with special characters in sentinel filename

- [ ] **Steps**: Synthetic injection with a tool name containing `/`, `:`, or `\`, e.g. `"tool_name": "weird/name:thing"`. Reset throttle first.
- [ ] **Expected:**
  - Sentinel file created with sanitized name (chars replaced with `_`)
  - No filesystem error
  - One warning logged

### 4.2 Concurrent hook invocations

- [ ] **Steps**: In rapid succession (or via parallel Claude Code sessions), trigger the same unknown tool from multiple processes simultaneously.
- [ ] **Expected:** No race condition errors. At most one warning per tool name.

### 4.3 Sentinel directory is read-only / unwritable

- [ ] **Steps**: Manually make `~/.claude/logs/.unknown_tool_warnings/` read-only, then trigger an unknown tool.
- [ ] **Expected:** Hook does NOT crash. Warning is silently skipped (best-effort throttling). Sesslog entry still written.

  Restore writability after this test:

  **PowerShell**:
  ```powershell
  attrib -R "$env:USERPROFILE\.claude\logs\.unknown_tool_warnings"
  ```

  **POSIX**:
  ```bash
  chmod u+w ~/.claude/logs/.unknown_tool_warnings
  ```

### 4.4 Existing logs from before v0.2.1 not modified

- [ ] **Steps**: Open an old `.sesslog_*.log` file from a session that ran on v0.2.0 or earlier.
- [ ] **Expected:** Pre-existing `{Skill: }` and `{PowerShell: }` empty entries remain (we don't rewrite history). Only NEW entries get the proper handlers and `?` markers.

  > **Note**: Per the CHANGELOG, original transcripts (`.jsonl` files in `~/.claude/projects/`) contain the actual command text — recovery is possible via the future gf-jumper tool (issue #23) when built.

## What the automated tests DON'T cover

These are the things this checklist exists to verify — mocks systematically miss them:

- **Real subprocess hook invocation**: pytest mocks don't actually run `python log-command.py` as a Claude Code subprocess. Cross-process sentinel throttling is only fully validated in real hook execution.
- **PowerShell vs Bash detection in real Claude Code**: Tests mock `tool_name="PowerShell"`. They don't validate that Claude Code on Windows actually emits `tool_name="PowerShell"` for PowerShell calls (vs `tool_name="Bash"` if Claude Code routes it differently).
- **File appearance under live `~/.claude/sesslogs/`**: Tests use temp dirs. The dedicated `.unknowns_*.log` file appearing in real session dirs requires manual verification.
- **Channel routing with merged user configs**: Unit tests verify `_default_channels()` and `_default_category_routes()` factory output. They don't verify ConfigLoader merge behavior end-to-end with a real user config file.
- **IDE autocompletion**: Schema correctness is validated structurally. The IDE actually surfacing autocomplete suggestions for `unknowns`/`unknown` requires manual visual confirmation in VS Code.
- **Visual readability of `?` marker**: Whether `{?ToolName: ...}` actually reads cleanly in a tail/grep workflow vs feeling cluttered. Mocks can't judge UX.
- **Cross-shell command behavior**: Tests run in pytest's Python; they don't validate the cmd.exe/PowerShell command snippets in this checklist actually work as written.
- **Performance impact of sentinel file checks**: One filesystem stat per hook call for unknown tools is presumably negligible; only real-world load can confirm.
- **Backwards compatibility for users with exotic configs**: Tests cover the typical merge case. Users with deeply customized `tool_overrides` plus `category_routes` plus disabled channels combinations are infinite — manual smoke test on representative configs is the only realistic guard.
