# Human Test Checklist: tools-output channel + TaskOutput template (v0.3.7-pre #87 P2a)

**Target version:** v0.3.7
**Ships with commit:** v0.3.7-pre: TaskOutput summary_template + .tools-output_* channel + sesslog cap -- P2a/#87
**Companion automated tests:** 371 total — 27 in `tests/one-offs/test_tasks_channel_coverage.py` covering TodoWrite routing, mcp_server_routes, TaskStop/TaskOutput overrides, TaskOutput content extraction, summary_template behavior, and tools-output channel pinning. Cross-cutting updates in `test_tools_channel.py` and `test_unknown_tool_routing.py` for the 9-channel default set.

## What this is

claude-session-logger ships per-channel log files (`.shell_*`, `.sesslog_*`, `.tools_*`, `.tasks_*`, ...) that capture Claude Code tool invocations. This commit changes how the `TaskOutput` tool (reading stdout from a backgrounded process) is recorded:

1. **TaskOutput dropped from `.shell_*`** — process-state queries aren't shell history
2. **TaskOutput now extracts `tool_response.task.output`** — was only logging `task_id` (useless)
3. **TaskOutput emits a `summary_template`** so per-channel `max_chars` actually truncates (without this, the `_legacy_complete` shortcut bypasses verbosity caps)
4. **New `.tools-output_*` channel** — disabled by default, opt-in like `.fileio_*`. Captures FULL process outputs when enabled
5. **New sesslog per-role cap** for `task-output` at 200 chars — kitchen sink stays focused on "what was invoked" with bounded preview
6. **`TaskStop` now extracts `tool_response.message`** — shows outcome like "Task 42 stopped successfully"

This checklist verifies behavior that mocks can't: cross-platform file naming with hyphens, live config-merge round-trips, reconciliation interactions with disabled channels, marker broadcast filtering, real multi-line process output legibility, and the experiential quality of the truncation levels.

2. **Confirm marketplace points at the local directory** (verifies you're testing the right code, not a stale install):

   **PowerShell**:
   ```powershell
   Get-Content $env:USERPROFILE\.claude\plugins\known_marketplaces.json | Select-String "dazzle-claude-plugins" -Context 0,4
   ```

   **POSIX**:
   ```bash
   grep -A 4 "dazzle-claude-plugins" ~/.claude/plugins/known_marketplaces.json
   ```

4. **Reset between sections**: clean the synthetic session dir between major test sections (the harness recreates it on next event):

   **PowerShell** (or POSIX bash):
   ```bash

   ```

### HV.6 — Channel file uses the literal hyphen in the basename

Validates: cross-platform file-naming for `.tools-output_*` (NTFS, ext4, APFS all allow hyphens but path-parsing code might split on `-`).

- [ ] **Steps:** after HV.5, look at the literal filename — it MUST contain the hyphen verbatim, not be split into separate path segments:

  **PowerShell**:
  ```powershell
  (ls $env:USERPROFILE\.claude\sesslogs\__eeeeeeee-test-taskoutput-87-eeeeeeeeeeee_$env:USERNAME\.tools-output*).Name
  ```

  **POSIX**:
  ```bash
  basename ~/.claude/sesslogs/__eeeeeeee-test-taskoutput-87-eeeeeeeeeeee_$USER/.tools-output*
  ```
- [ ] **Expected:** name is `.tools-output_<shell>__...<id>_<user>.log` — single file, hyphen preserved between `tools` and `output`.
- [ ] **Why this matters:** the discovery regex in `reconciliation.py:discover_channel_basenames` uses `[\w-]+` to capture the basename. If we accidentally created a literal `.tools` + a subdirectory `output_*`, reconciliation would silently mis-track the file on session rename.

## Section 6: Real-world TaskOutput legibility (qualitative)

The synthetic test uses fake `[next-dev]` lines. Real-world TaskOutput will have ANSI color codes, real timestamps, real stack traces, JSON blobs, etc.

- [ ] **Prerequisite:** a real project where `npm run dev` (or any long-running dev server) runs
- [ ] **Steps:** in a NEW Claude Code session, ask Claude to run that dev command in the background, wait 30 seconds, then read its output. The model will issue real `Task*` + `TaskOutput` calls.
- [ ] **Check:** look at the resulting `.sesslog_*` and `.tools_*` entries for TaskOutput. Do they:
  - [ ] Render readably (no broken character sequences from ANSI codes)?
  - [ ] Show meaningful context in the truncated portion (not just a useless prefix)?
  - [ ] Make it obvious WHICH task's output it is (`#<task_id>` prefix visible)?
- [ ] **Why this matters:** this is the experiential quality check. The synthetic test verifies mechanics; this verifies usefulness.

### 6.2 Multi-line output in tools-output channel (if enabled)

- [ ] **Steps:** as 6.1, but with tools-output enabled in your real config
- [ ] **Check:** open `.tools-output_*.log` in a text editor (not `cat` since that respects existing newlines anyway):
  - [ ] Multi-line output renders as actual newlines (RENDER policy), not as literal `\n` escape sequences
  - [ ] ANSI color codes either render in a color-capable terminal or appear as readable text (not garbled)
- [ ] **Why:** `tools-output` defaults to `NewlinePolicy.RENDER` specifically so it's readable as a log stream rather than grep-friendly single lines

## What the automated tests DON'T cover

Explicit gaps that only manual testing can verify:

- **Cross-platform file naming** — automated tests use `tmp_path` (typically POSIX-style); real Windows NTFS behavior with hyphens in basenames is verified here
- **Live config-merge round-trip** — automated tests call `apply_override_config` directly; real `~/.claude/plugins/settings/session-logger.json` loading is verified here (HV.5, 7.x)
- **Reconciliation × disabled channel interaction** — automated tests don't exercise the rename path against the new disabled channel (Section 3)
- **Marker broadcast filtering** — automated tests verify marker broadcast to enabled channels but not the "disabled channels stay silent" inverse (Section 4)
- **Real subprocess output legibility** — synthetic harness uses fake dev-server text; real ANSI codes, error stacks, and JSON blobs only happen in production (Section 6)
- **Editor / tail rendering** — does `.tools-output_*.log` actually read well when tailed in a terminal vs opened in VS Code vs grepped? Mocks can't verify "feels good to use"
- **Disk-pressure under high TaskOutput volume** — if a user enables `tools-output` and a process produces gigabytes, do we gracefully handle it? Not in scope but worth one observation
