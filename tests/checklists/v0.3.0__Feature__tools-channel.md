## What this is

End-to-end verification that v0.3.0 actually creates `.tools_*.log` in real session directories, that the channel captures the right tools, and that customized configs auto-pick up the new defaults via per-key merge. Items mocks can't cover: real subprocess hook invocation, file appearance on disk, IDE schema autocomplete, multi-channel write atomicity.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact behaviors v0.3.0 introduces. If any fail, the work is broken.

### HV.1 — `.tools_*.log` file is created in a fresh session

Validates: new channel actually fires.

- [ ] **Steps**: Start a new Claude Code session in any project directory. Run any tool (e.g., ask Claude to list files via `LS` or run a `Bash` command).
- [ ] **Then check**:

  **PowerShell**:
  ```powershell
  Get-ChildItem "$env:USERPROFILE\.claude\sesslogs\*\.tools_*"
  ```

  **POSIX**:
  ```bash
  ls ~/.claude/sesslogs/*/.tools_*
  ```

- [ ] **Expected**: A `.tools_<context>.log` file exists in the active session directory.
- [ ] **Why this matters**: If the file isn't created, the entire feature is non-functional.

### HV.2 — Tools channel captures tool calls

Validates: routing actually sends entries here.

- [ ] **Steps**: After triggering a few tool calls (Bash, Read, Edit), inspect the new file:

  **PowerShell**:
  ```powershell
  Get-Content "$env:USERPROFILE\.claude\sesslogs\<your-session-dir>\.tools_*" -Tail 10
  ```

  **POSIX**:
  ```bash
  ls -t ~/.claude/sesslogs/*/.tools_* | head -1 | xargs tail -10
  ```

- [ ] **Expected**: Recent entries match what you triggered, e.g.:
  ```
  [[2026-05-05 12:34:56]] {Bash: ls -la }
  [[2026-05-05 12:35:01]] {Read: "/path/to/file" }
  ```
- [ ] **Why this matters**: Confirms routing works, not just file creation.

### HV.3 — Same entries also appear in `.sesslog_*.log` (no regression)

Validates: backwards compatibility — sesslog still kitchen sink.

- [ ] **Steps**: Compare entry counts:

  **PowerShell**:
  ```powershell
  $tools = Get-Content "$env:USERPROFILE\.claude\sesslogs\<dir>\.tools_*" | Measure-Object -Line
  $sesslog = Get-Content "$env:USERPROFILE\.claude\sesslogs\<dir>\.sesslog_*" | Measure-Object -Line
  Write-Host "tools: $($tools.Lines) sesslog: $($sesslog.Lines)"
  ```

  **POSIX**:
  ```bash
  wc -l ~/.claude/sesslogs/<dir>/.tools_* ~/.claude/sesslogs/<dir>/.sesslog_*
  ```

- [ ] **Expected**: `.sesslog_*` has at least as many lines as `.tools_*` (sesslog includes tools + unknowns + future prose).
- [ ] **Why this matters**: Verifies the tools channel is additive — sesslog still receives the entries.

### HV.4 — Unknown tools do NOT appear in `.tools_*.log`

Validates: `unknown` route correctly excludes tools channel.

- [ ] **Steps**: Trigger an unknown tool (e.g., from synthetic injection — see Section 1.3 of the v0.2.1 checklist) OR find a recent `{?<ToolName>:` line from real Claude Code activity.
- [ ] **Then verify** the unknown entry is NOT in `.tools_*`:

  **PowerShell**:
  ```powershell
  Select-String -Pattern '\{\?' -Path "$env:USERPROFILE\.claude\sesslogs\*\.tools_*"
  ```

  **POSIX**:
  ```bash
  grep '{?' ~/.claude/sesslogs/*/.tools_*
  ```

- [ ] **Expected**: No matches in `.tools_*`. The unknown entry should ONLY appear in `.sesslog_*` and `.unknowns_*`.
- [ ] **Why this matters**: Tools channel must stay focused on known/categorized AI activity for it to be useful as the "find exact tool calls" view.

### HV.5 — `.shell_*.log` still clean of non-shell entries

Validates: no regression from v0.2.1 — shell channel stays curated.

- [ ] **Steps**: Confirm `Read`, `Edit`, `Skill`, etc. do NOT appear in `.shell_*.log`:

  **PowerShell**:
  ```powershell
  Select-String -Pattern '\{(Read|Edit|Skill|Glob|Grep):' -Path "$env:USERPROFILE\.claude\sesslogs\*\.shell_*"
  ```

  **POSIX**:
  ```bash
  grep -E '\{(Read|Edit|Skill|Glob|Grep):' ~/.claude/sesslogs/*/.shell_*
  ```

- [ ] **Expected**: No matches. `.shell_*` should contain only `Bash`, `PowerShell`, and other shell-style commands.
- [ ] **Why this matters**: Channel philosophy preservation across versions.

### HV.6 — Customized config still gets `tools` channel via merge

Validates: ConfigLoader's per-key merge picks up new defaults.

- [ ] **Steps**: Create or open `~/.claude/plugins/settings/session-logger.json` with a customized routing block missing `tools`:

  ```json
  {
    "routing": {
      "channels": {
        "shell": {"file_prefix": ".shell_", "enabled": true},
        "sesslog": {"file_prefix": ".sesslog_", "enabled": true},
        "tasks": {"file_prefix": ".tasks_", "enabled": true},
        "unknowns": {"file_prefix": ".unknowns_", "enabled": true}
      },
      "category_routes": {
        "_default": ["shell", "sesslog"],
        "task": ["shell", "sesslog", "tasks"]
      }
    }
  }
  ```

  Trigger any tool. Verify `.tools_*.log` still appears (defaults merged in for missing keys).

- [ ] **Expected**: `.tools_*.log` exists despite no explicit `tools` channel/route in user config. Per-key merge fills in defaults.
- [ ] **Why this matters**: No manual migration needed for existing users — the upgrade is transparent.

## Section 1: Routing edge cases

### 1.1 Tool overrides take precedence over category routes

- [ ] **Steps**: Add a `tool_overrides` entry that excludes `tools`:
  ```json
  { "routing": { "tool_overrides": { "Bash": ["shell", "sesslog"] } } }
  ```
- [ ] **Expected**: Bash entries appear in `.shell_*` and `.sesslog_*` but NOT `.tools_*`. Tool overrides win over the new default route.

### 1.2 Disabling tools channel via enabled flag

- [ ] **Steps**: Set `routing.channels.tools.enabled: false` in config. Trigger tools.
- [ ] **Expected**: No `.tools_*` file is written for that session. Other channels unaffected.

### 1.3 Disabling via route override

- [ ] **Steps**: Set `routing.category_routes._default: ["shell", "sesslog"]` (no tools).
- [ ] **Expected**: `.tools_*` not written. (Channel still enabled but nothing routes to it.)

## Section 2: Multi-tool sessions

### 2.1 Long session content distribution

- [ ] **Steps**: Run a session with mixed activity for 5-10 minutes (Bash, Read, Edit, TaskCreate, Skill invocations).
- [ ] **Inspect** all four channels:
  - `.shell_*` — should contain only Bash/PowerShell entries
  - `.tools_*` — should contain Bash + Read + Edit + TaskCreate + Skill entries
  - `.sesslog_*` — should contain everything (kitchen sink)
  - `.tasks_*` — should contain only Task* entries

- [ ] **Expected**: Each channel reflects its purpose; no entries are duplicated unexpectedly.

## What the automated tests DON'T cover

- **Real subprocess hook invocation** — pytest mocks don't run `python log-command.py` as a Claude Code subprocess
- **File appearance under live `~/.claude/sesslogs/`** — only manual verification via real Claude Code session
- **IDE autocompletion** — schema correctness verified structurally; actual VS Code IntelliSense behavior requires manual confirmation
- **Multi-channel write atomicity** — when an entry routes to 4 channels, all writes complete before next event (no partial state)
- **Performance impact** — adding a 4th file write per event has measurable cost; acceptable in practice but unmeasured
