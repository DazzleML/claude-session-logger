## What Phase 2+3 changes

End-to-end LogEntry/formatter cutover. The Channel Options Framework comes to life:
- Handlers stop emitting pre-formatted strings; they emit `LogEntry` objects with `raw_content` + optional rich-format `summary` template
- A `cclogger/formatters/` package replaces the single `formatters.py` module
- `format_for_channel()` dispatch routes each entry through the formatter named in `ChannelOptions.formatter`
- 4 hardcoded sites removed: 2 `truncate_preview(prompt, max_len=200)` literals in conversation.py, the `if channel_name == "tasks"` dispatch branch in logger.py, and the per-tool `truncate_preview` calls in Write/Edit handlers (rich-format snippets now substituted per channel verbosity)
- Per-channel default `ChannelOptions` baked in: `convo` uses `chat` formatter + RENDER + full; `sesslog` uses full; `tasks` uses `task-only`; `shell`/`tools`/`unknowns` use 100-char snippets (was 20)

This is the **first phase that produces intentional non-byte-identical output**. The new baseline at `tests/snapshots/v036_baseline/` reflects the expected new shape.

## What mocks don't cover (only this checklist catches)

- **Plugin install path correctness** — the marketplace artifact must include the new `cclogger/formatters/` subpackage (6 files). If the package is missing on disk, hooks will silently fail on import.
- **Real Claude Code event payloads** — synthetic_events.py uses fixture data; real PostToolUse/UserPromptSubmit events have field name variants we infer (`user_prompt` vs `prompt` vs `user_input` for example) — only a real session catches mismatches.
- **Multi-line USER prompt rendering in actual log files** — automated tests verify the formatter logic; this catches whether terminals + log viewers display the multi-line entries readably.
- **Cross-shell concerns** — Bash, PowerShell, cmd.exe behave differently for stdin encoding, path separators, and process spawn. Only invoking from each shell catches issues.
- **`~/.claude/logs/hook-debug.log` warnings** — automated tests don't fail on warnings; only manual eyeball spots ImportWarning, DeprecationWarning, or unexpected content there.
- **Cache invalidation timing** — the plugin install caches; if a version conflict exists, the cached version may serve stale code. Manual restart + cache verification catches this.

## Addendum: framework completeness additions (post-original-draft)

After the original checklist was written, Phase 2+3 grew to include the framework-completeness work that closed two gaps: (1) channels couldn't express "full default with per-role overrides," and (2) the kitchen-sink sesslog channel showed full Write/Edit content which was unreadable. Sections below cover the resulting changes. The meta-lesson worth carrying forward into future checklists:

> **When the design says X and the resulting log is unusable, the design is wrong.** Section 2.2 of this checklist originally documented "FULL content on sesslog" as expected — a usability regression that real-session eyeballing caught after automated tests already passed. Future checklists should always include a "open the log files and read them" pass, not just "does it match the spec" pass.

### Section 2.5: Sesslog readability after Write/Edit (regression fix verification)

Section 2.2 (Edit FULL content on sesslog) **was a regression in disguise**. The fix uses the new `_default` keyword to truncate Write/Edit on sesslog while keeping prose full.

- [ ] In a fresh session, do a `Write` of a >200-char body to a new file. Open `.sesslog_*.log`. Verify the entry shows `"<path>" ← "<20-char preview>..." (NL) }` — NOT the full body. (v0.3.6 readability restored.)
- [ ] In the same session, do an `Edit` replacing a multi-line block. Verify sesslog entry shows `"<path>:<line>" ← "<20-char preview>..." (-N/+ML) }` — NOT the full new_string.
- [ ] Send a USER prompt of >100 chars. Verify sesslog still shows the FULL prompt body (`{USER: <full prose> }`) — the `_default = "full"` keeps prose intact.
- [ ] Open `.shell_*.log` and `.tools_*.log` — Write/Edit there show 100-char snippets per their channel options (unchanged from earlier Phase 2+3 behavior).

### Section 11: `.fileio_*` opt-in workflow

The new `.fileio_*` channel captures full file-I/O content. Disabled by default.

- [ ] Fresh session — confirm `.fileio_*.log` does NOT exist in the sesslog directory.
- [ ] Add config at `~/.claude/plugins/settings/session-logger.json`:
  ```json
  {
    "routing": {
      "channels": {
        "fileio": {"file_prefix": ".fileio_", "enabled": true}
      }
    }
  }
  ```
  Restart Claude Code. Confirm `~/.claude/logs/hook-debug.log` shows no errors.
- [ ] Do a `Write` and an `Edit` in a session. Confirm `.fileio_*.log` appears AND contains:
  - The FULL content (no truncation, no `...`)
  - Real newlines visible (RENDER policy — multi-line entries)
  - `{Write:` and `{Edit:` braces wrapping each entry
- [ ] Do a `Read`. Confirm Read appears on `.fileio_*.log` too (Read recategorized to `io`).
- [ ] Disable fileio in config (`"enabled": false`), restart, verify no new entries land in `.fileio_*.log`.

### Section 12: `_default` per-role-dict fallback (framework feature)

Custom config with the new keyword. Verifies the framework extension end-to-end.

- [ ] Create config:
  ```json
  {
    "routing": {
      "channels": {
        "shell": {
          "file_prefix": ".shell_",
          "options": {
            "verbosity": {"_default": "full", "edit": {"max_chars": 50}}
          }
        }
      }
    }
  }
  ```
  Restart. Do an Edit and a Bash. Confirm:
  - `.shell_*.log` Bash entry shows full content (matches `_default = "full"`)
  - `.shell_*.log` Edit entry shows 50-char snippet + `...` (matches `edit` override)
- [ ] Try an intentionally-wrong config mixing hint + role keys:
  ```json
  {"options": {"verbosity": {"max_chars": 50, "user": "full"}}}
  ```
  Restart. Check `~/.claude/logs/hook-debug.log` — should see a warning about mixed hint+role keys.

### Section 13: Backward compat — Read recategorization

Read moved from `system` to `io`. Users with customized `category_routes` may see different behavior.

- [ ] If you have a custom config setting `routing.category_routes.system`, Read no longer routes via that key. Add Read to your `io` route OR use `tool_overrides.Read` if you want Read on different channels than other file-I/O.
- [ ] Verify default config: `Read` entries appear on `.shell_*`, `.sesslog_*`, `.tools_*` (the `io` route minus `fileio` since fileio is disabled). No regression.
- [ ] `system` category now contains only plan modes — verify `EnterPlanMode`/`ExitPlanMode` still log correctly (single-line entries on shell/sesslog/tools per `_default` route).

### Section 14: Subtype channel inheritance (framework completeness)

When subtype routing is enabled, derived channels inherit parent options.

- [ ] Add config: `{"routing": {"subtype_routing": {"bash": true}}}`. Restart. Run a `Bash` AND a `PowerShell` command.
- [ ] Verify `.shell-bash_*.log` and `.shell-powershell_*.log` exist
- [ ] Verify they use the parent `shell` channel's `max_chars=100` (not the global 20 — Bash entries are short here so this is mostly inert; the contract matters for longer commands)
- [ ] Explicitly declare a subtype channel with custom options:

  Verify `.shell-powershell_*.log` now uses 50-char (the explicit declaration wins over parent inheritance).

### Section 15: Real-session readability pass (the "actually read the logs" check)

The meta-lesson from the regression we caught — automated tests can't tell you whether a log is readable.

- [ ] Run a typical 10-15-minute session that includes: 2-3 user prompts, 1-2 Write operations on real source files (≥100 lines), 3-5 Edit operations, several Bash commands, possibly a Task subagent invocation.
- [ ] Open `.sesslog_*.log`. Scroll through. Answer: "If I needed to find a specific event from this session 3 hours later, could I grep + scan productively?"
  - GREEN: USER/AI prose is readable, tool calls are visible, no single entry dominates >5 lines, file paths + line numbers are clickable.
  - YELLOW: One channel's verbosity setting needs tweaking; nothing fundamentally broken.
  - RED: A single Write/Edit entry produces 50+ lines, OR prose is truncated to `...`, OR tool entries lack vim-jumpable paths.
- [ ] Open `.shell_*.log`. Confirm it's still copy-pasteable as a shell history (no LogEntry artifacts, no Markdown, just `command` lines).
- [ ] Open `.tools_*.log`. Confirm it's the "AI activity" view — tool calls without prose, rich Edit format preserved.
- [ ] Open `.convo_*.log`. Confirm it reads as a chat transcript: USER/AI/AGENT entries, real newlines, full prose.
- [ ] Open `.tasks_*.log`. Confirm task-only format: `[[ts]] {CREATE #N: subject | description }` style.
- [ ] If `.fileio_*` is enabled: open it, confirm it has full file content (multi-line) for Write/Edit and path-only entries for Read.
